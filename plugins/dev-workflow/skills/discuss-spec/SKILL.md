# discuss-spec — 仕様の壁打ち

会議や資料がなくても、その場の対話で仕様を決めるための skill。/reflect-decisions(人の決定 → doc)の**上流**に位置し、「決定そのもの」を壁打ちで作る。成果物は決定録のみで、doc への反映は /reflect-decisions のパイプライン(精査・出典・タグ付け)に委ねる。

## 原則

1. **聞き役に徹しない**。論点ごとに現状・選択肢・トレードオフ・推奨案を必ず用意して打ち返す(壁打ち相手であって議事係ではない)。ただし誘導はしない — ユーザーの発言に無いことを決定に含めない
2. **範囲を守る**。合意した論点リストの範囲だけを決める。対話中に話が広がったら「スコープ外の論点」として記録し、別セッションを提案する(発散防止)
3. **既決を蒸し返さない**。doc に `[決]`/`[実]` で記載済みの事項・ADR 済みの判断は前提として扱う。ユーザーが変更を望む場合は「現行決定の変更」であることを明示し、現行との差分・影響を添えて扱う
4. **矛盾の即時指摘**。回答が既決事項・既存実装・同セッション内の他の決定と矛盾したら、先へ進む前にその場で指摘して解消する
5. **無理に決めさせない**。判断材料が足りない論点は「未決(何が分かれば決められるか+確認先)」として閉じてよい。結論は 決定 / 暫定 / 未決 / 却下案 に峻別する
6. **doc は直接書かない**。書くのは決定録のみ。反映は /reflect-decisions へチェーンする(反映ロジックの二重実装をしない)

## 引数

| 引数 | 内容 |
|---|---|
| テーマ(自由文) | 壁打ちしたい範囲(例: 「認証まわり」「スケジュールと見積もり」「画面遷移」「非機能要件」) |
| (省略時) | doc の未確定箇所(「（未定）」・ガイド文・索引の 🔴/🟡)を走査し、壁打ち候補を優先度付きで提示して選んでもらう。ほぼ全文書が空の新規なら「01 目的 → 03 要件 → 07 計画」の順の全体壁打ちを提案 |
| `--doc=<番号>` | 対象文書を絞る(例: `--doc=03`) |
| `--quick` | 主要論点のみに絞り、1 論点 1 質問を目安に短時間で回す |

## Phase 1: 現状把握(テーマ → 文脈)

1. `.claude/project-profile.yml` があれば Read(`root` / `source_of_truth`)。doc ディレクトリを検出(`doc/` → `docs/`)。統一構成が無ければ /init-project の先行実行を提案する(反映先が無いため)
2. テーマから関連文書・節を特定する(対応は /reflect-decisions の反映マップと同じ: 目的・スコープ → 01 / 要件 → 03 / 設計判断 → 04 / 運用 → 05 / 計画・期限 → 07)
3. 関連する既存記述(既決・未定)・ADR・**既存実装の現状**(既存プロジェクトでは対象領域のコード・スキーマ・ルーティングを実測)を読み、「決まっていること / 空欄のこと / 実装が既にあること」を整理する
4. モード判定: 関連文書がほぼガイド文のまま → **新規モード**(ゼロから決める)。記述あり → **差分モード**(既存を前提に追加・変更を決める)

## Phase 2: 論点分解とスコープ合意

1. テーマを「決めるべき論点」に分解する(例: 認証 → 方式 / プロバイダ / セッション管理 / 権限モデル / 未ログイン時挙動)。各論点に現状(doc・実装)と決定の反映先文書を添える
2. 論点リストを推奨順(依存関係順: 前の決定が後の選択肢を狭めるものを先に)で提示し、**AskUserQuestion(multiSelect)で今回扱う範囲を合意**する。`--quick` は主要論点に絞る
3. 論点が多い場合(目安 8 超)は分割を提案する(今回はここまで、残りは次セッション)

## Phase 3: 壁打ちループ(1 論点ずつ)

1. 論点ごとに提示する: **背景**(現状の doc・実装)→ **選択肢 2〜4 案**(それぞれのトレードオフ)→ **推奨案+理由**
2. 進め方は論点の性質で使い分ける:
   - 選択式が向く論点(技術選定・方式) → AskUserQuestion(推奨に (Recommended) を付ける)
   - 発散が必要な論点(目的・スコープ・業務フロー) → 自由対話で深掘りし、出た内容を復唱して合意を確認する
3. 回答を受けたら: 矛盾チェック(原則 4)→ 問題なければ**決定を 1 行に整形して復唱**し、次の論点へ。迷いが残る回答は「暫定」、材料不足は「未決」として閉じる
4. 進捗を随時可視化する(N 論点中: 決定 M / 暫定 K / 未決 L)。**中断可能** — 途中でも Phase 4 の決定録は生成できる

## Phase 4: 決定録の生成

1. セッション結果を決定録 MD にまとめる:
   - メタ: 日付 / テーマ / 扱った論点(合意したスコープ)
   - **決定**: 内容(1 行 1 決定)+ 理由 + 却下した代替案(比較検討したもの)
   - **暫定**: 内容 + 確定の条件
   - **未決(宿題)**: 何が分かれば決められるか + 確認先・期限(あれば)
   - スコープ外に出た論点(次セッション候補)
2. 形式は /reflect-decisions の判定基準(決定 / 暫定 / 未決 / 却下)に合わせる — そのまま入力になる
3. 保存先: 既存の議事録・資料置き場があればそこへ。無ければ `doc/minutes/YYYY-MM-DD_壁打ち_<テーマ>.md` を提案する(reflect-decisions が出典リンクとして参照する)

## Phase 5: 反映チェーン

1. 「**/reflect-decisions で doc へ反映しますか?**」を AskUserQuestion で確認する
   - Yes → 決定録を入力にチェーン実行する(決定録は構造化済み・矛盾解消済みのため、精査は軽量に通ることが多い)
   - No → 決定録のパスと、後から `/reflect-decisions <決定録パス>` で反映できることを案内する
2. 反映後の導線(実装タスク化 → /create-task、クライアント提出 → /export-doc)は reflect-decisions 側のチェーンに委ねる

## 最終ゲート(完了報告前セルフチェック)

- [ ] 合意した論点リストの範囲だけを扱った(広がった話はスコープ外として記録した)
- [ ] すべての論点で選択肢+トレードオフ+推奨を提示した(聞くだけの進行をしていない)
- [ ] ユーザーの発言に無い決定を作っていない(推奨はしても、採用はユーザーの回答で確定した)
- [ ] 既決事項・既存実装との矛盾を放置したまま決定にしていない
- [ ] 決定 / 暫定 / 未決 / 却下を峻別して決定録に残した
- [ ] doc/ を直接書き換えていない(反映は /reflect-decisions に委ねた)
- [ ] 決定録の保存先を確認し、反映チェーン(または後から反映する方法)を案内した

## 関連 skill

- 前提: /init-project(doc/ の統一構成)。既存プロジェクトでは /understand-project で把握してからだと論点の精度が上がる
- 後続: /reflect-decisions(決定録 → doc 反映)→ /create-task → /do-task → /update-doc --task
- 使い分け: 会議・資料が**既にある**なら本 skill を挟まず /reflect-decisions を直接使う。決まっていないことを**これから決める**なら本 skill
