# stack-research — バージョン固有の注意点調査とノート生成

プロジェクトの依存を検出し、**その実バージョンに固有の**アンチパターン・ベストプラクティス・セキュリティ注意点を Web 調査して、`doc/06_stack-notes.md` に生成・更新する。汎用知識の書き写しではなく「このプロジェクトのこのバージョンで、やってはいけないこと・推奨されること」だけを、出典付きで置くのが目的。

## 原則

1. **バージョンを特定してから調べる**。「React の注意点」ではなく「React 19.x の注意点」を調べる。実バージョンは lockfile の解決済みバージョンを正とする(マニフェストの range ではない)
2. **全項目に出典(URL +確認日)**。出典のない伝聞・記憶ベースの記述を書かない。調べて見つからなければ「特記なし」と明記する(それも有益な結果)
3. **公式優先**: 公式ドキュメント / リリースノート / GitHub Security Advisories → 信頼できる技術記事の順。対象バージョンと無関係な(古い・新しすぎる)情報を混ぜない
4. **コードは変更しない**。成果物はノートと索引追記のみ。問題が実在したら /create-task を提案する
5. 機密ファイル(profile の `secret_paths`)は読まない

## いつ実行するか

- /init-project 直後(最初の 1 回)
- 依存の追加・更新後(`--update` で差分のみ)
- セキュリティ勧告のニュースを見たとき(`--security-only` で高速確認)

## 引数

| 引数 | 役割 |
|---|---|
| (なし) | 主要依存を全調査して生成(既存ノートがあれば更新) |
| `--update` | 差分更新: バージョンが変わった・新規追加された依存のみ再調査(不変の依存の記述は保持) |
| `--security-only` | セキュリティ勧告・CVE の確認のみ高速に実行 |
| `--deps=<pkg1,pkg2>` | 指定依存のみ調査 |

## Phase 1: 依存とバージョンの検出

1. profile を解決(`root` / `has_code` / `secret_paths`)。`has_code: false` なら対象外と報告して終了
2. マニフェスト・lockfile から**主要依存を選定**する(目安 8〜15 個):
   - 言語/ランタイムのバージョン(Node: engines / .node-version / volta、Python: requires-python、PHP / Go / Rust)
   - フレームワーク(next / react / vue / hono / express / laravel / django 等)
   - ORM・DB クライアント / 認証 / 決済 / ファイルアップロード等の「書き方を誤ると事故るもの」を優先
   - ビルド・テストツール(vite / vitest / playwright / biome 等)は影響が大きいもののみ
3. **実バージョンは lockfile から解決済みのものを取る**。パッケージマネージャのコマンド(`pnpm list --depth 0` 等)が使えるなら併用する
4. 検出結果(依存 × 実バージョンの表)を提示してから調査に進む

## Phase 2: Web 調査(並列 fan-out)

依存ごと(または密接な関連グループごと: 例 next+react)に調査サブエージェント(general-purpose)を**単一メッセージで並列起動**し、WebSearch / WebFetch で調べさせる:

1. **バージョン固有アンチパターン**: この major/minor でやってはいけない書き方、旧バージョンの癖で書きがちな誤り(非推奨 API・変更されたデフォルト・移行済みの作法)
2. **バージョン固有ベストプラクティス**: このバージョンで公式が推奨する書き方・構成・パフォーマンスパターン(新しく使えるようになった推奨機能、旧来手法のより良い代替を含む)
3. **セキュリティ**: GitHub Security Advisories / 公式勧告 / CVE。**検出バージョンが影響範囲に入るか**、修正バージョンはどれか
4. **この構成での定番ミス**: 公式セキュリティガイドから、このプロジェクト構成に関係するもの(認可チェック漏れ・入力検証・secrets の露出・SSRF/XSS 等)

各エージェントへの指示に含めること: 対象の依存名と**実バージョン**、返答形式(アンチパターン / ベストプラクティス / セキュリティ / 特記なし項目 / 出典 URL +確認日)、「見つからない場合は『特記なし』と返す(推測で埋めない)」、**「出典に挙げる URL は WebFetch で本文を実際に開いて内容を確認してから記載する(検索結果のスニペットや記憶だけで書かない。本文確認できなかった情報は採用しない)」**。

## Phase 3: ノート生成

`doc/06_stack-notes.md`(`docs/` を使うプロジェクトは docs/ 配下)に以下の構成で Write する:

```markdown
# 06. スタックノート(バージョン固有の注意点)

> 生成: {日付} / 検出元: {lockfile 名}(hash: {`git hash-object <lockfile>` の先頭 12 桁}) / 更新: /stack-research --update

## 検出した主要依存
| パッケージ | 実バージョン | 備考(EOL・サポート状況) |

## バージョン固有のアンチパターン
### {依存名} {バージョン}
- ❌ {やってはいけない書き方} → ✅ {正しい書き方}({出典})

## ベストプラクティス
### {依存名} {バージョン}
- ✅ {このバージョンで推奨される書き方・構成}({出典})

## セキュリティ注意点
### 既知の脆弱性
| 依存 | 勧告 / CVE | このバージョンへの影響 | 対応(修正版 / 回避策) |
### この構成での定番ミス
- {観点と対策}({出典})

## アップグレード注意
- {次メジャーへの既知の破壊的変更}({出典})

## 出典
- {URL}(確認日: {日付})
```

- ヘッダの lockfile ハッシュは**鮮度検出用**: /understand-project がこの値と現在の lockfile を比較し、依存が変わったのにノートが古い状態を警告する(git が無い環境ではハッシュを省略し日付比較にフォールバック)
- `doc/README.md`(索引)があれば 06 の行を**番号順を保つ位置(05 と 07 の間)**に挿入する
- 既存ノートがある場合、バージョンが変わっていない依存の記述は保持し、変わったものだけ差し替える(全消し再生成しない)
- ノートを生成・更新したら `.claude/grasp.md` があれば削除する(⚠️ 事項が変わったため、次回の /understand-project が再把握して反映する)

## Phase 4: 実在チェックとタスク化の提案(推奨)

1. 生成したアンチパターンのうち機械検出可能なもの(非推奨 API 名・危険な関数・脆弱バージョンの import)を Grep でコードスキャンし、該当があればノートの当該項目に「⚠️ 本プロジェクトに該当あり: {path}:{line}」を追記する
2. **該当があった場合、重要度順の一覧を提示して AskUserQuestion で「タスク化しますか?」を確認する**(セキュリティ該当 = 高、非推奨 API = 中、ベストプラクティス乖離 = 低。multiSelect でどれを対象にするか選べる形に)
3. **Yes → /create-task にチェーン実行**する。入力として渡すもの: 選択された該当箇所(path:行)+ノートの該当節(出典付きの根拠)+種別ヒント(脆弱性対応 = 依存更新を伴う機能変更 / 非推奨 API・アンチパターン置換 = `--refactor` 相当の挙動維持)。複数該当を 1 タスクにするか分けるかは create-task の分割判断に委ねる
4. No → ノートの ⚠️ 記録だけ残して終了(後からノートを見て /create-task できる)。**このスキル自身では修正しない**

## 最終ゲート(出力前セルフチェック)

- [ ] すべての記述に出典(URL +確認日)がある、または「特記なし」と明示した
- [ ] 実バージョンを lockfile から取り、バージョン不一致の情報を混ぜていない
- [ ] セキュリティ項目は「このバージョンが影響を受けるか」まで判定した
- [ ] コードを変更していない(ノートと索引のみ)
- [ ] 実在チェックの結果(該当あり/なし)を報告した

## 関連スキル

- 生成したノートの参照元: /understand-project(⚠️ 事項に反映)・/create-task(技術的考慮事項。--refactor ではアンチパターン検出対象に追加)・/data-audit(バージョン固有の脆弱性・注意点を監査観点に追加)
- 該当箇所の修正: /create-task → /do-task
- 初期導入: /init-project の直後に実行を推奨
