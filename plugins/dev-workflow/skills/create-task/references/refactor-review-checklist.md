# リファクタリング レビューチェックリスト(6 カテゴリ)

SKILL.md「Phase 6: レビュー」の詳細版。小・中規模は team-lead が、大規模は reviewer(Explore, 読み取り専用)がこのチェックリストで検証する。

**指摘は盲信しない。** team-lead が各指摘を実コードで裏取りし、valid / invalid / needs-user にトリアージする。false positive は理由を記録する。React 固有項目は該当スタックのみ適用し、他スタックは読み替える。

## 1. 機能保全(最重要)

- [ ] リファクタ前後で外部から見た挙動が同一か
- [ ] 公開契約(公開 API / props / 関数シグネチャ / エンドポイント / 返却値の形)に破壊的変更がないか
- [ ] import / 参照パスが正しく更新されているか
- [ ] 参照元(呼び出し側)が全て更新されているか(`find_referencing_symbols` / Grep で網羅確認)
- [ ] 契約変更が必要だった場合、別変更として切り出し、影響範囲とテスト更新が明示されているか

## 2. コード品質

- [ ] 分割後の各ファイル/関数が適切なサイズか(過度な断片化＝50 行未満への無意味な分割をしていないか)
- [ ] 命名がプロジェクト規約に沿っているか
- [ ] 型が適切か(新たな any を導入していないか、型が具体的か。既存の any が残るのは許容)
- [ ] デッドコード(未使用の import / 変数 / 関数、到達不能コード)が残っていないか
- [ ] 重複が実際に解消されたか(DRY 違反の再発なし)
- [ ] エラーハンドリングの方針が既存と一貫しているか

## 3. アーキテクチャ・依存関係

- [ ] 層/責務の分離が適切か(表示・状態副作用・純粋ロジックが混ざっていないか)
- [ ] 依存方向が正しいか(循環依存がない、外→内の一方向)
- [ ] 共通化が過度な抽象化になっていないか(引数が多すぎる共通部品等)
- [ ] composition(小さな部品の組み合わせ)が正しく機能しているか
- [ ] (React) Server / Client Component 分離が適切で、`"use client"` が末端最小か

## 4. ディレクトリ構成

- [ ] 移動後の import パスが全て更新されているか
- [ ] 移動元に旧ファイル・空ディレクトリが残っていないか
- [ ] 関連ファイルが近接配置(colocation)され、機能凝集度が上がったか
- [ ] 構成パターンが機能間で統一されているか
- [ ] 過度なネスト(不要な深い階層)が発生していないか
- [ ] barrel(index の再エクスポート)の有無が基準に沿っているか

## 5. テスタビリティ

- [ ] 抽出した関数が純粋か(外部状態に依存せず、同じ入力に同じ出力)
- [ ] 関数のシグネチャが明確か(引数・戻り値の型が具体的)
- [ ] 表示層に計算・変換ロジックが残っていないか
- [ ] 副作用が Hook / サービスに適切に分離されているか
- [ ] 定数が適切に外部化されているか
- [ ] (バックエンド) ドメインロジックが I/O から切り離され、単体テスト可能か

## 6. プロジェクト規約

- [ ] ファイル命名規則(既存パターン)に従っているか
- [ ] ディレクトリ配置規約に従っているか
- [ ] import パス規約(エイリアス等)に従っているか
- [ ] 権威参照(CLAUDE.md / doc / メモリ)の禁止事項・地雷に抵触していないか
- [ ] フレームワーク/スタック固有の推奨パターンに沿っているか

## 追加観点(大規模・パフォーマンス懸念時)

- [ ] 不要な再計算・再生成を招いていないか(React: React.memo / useMemo / useCallback の適用余地)
- [ ] データアクセスの非効率(N+1 クエリ、不要な取得)を生んでいないか
- [ ] API 呼び出しの非効率がないか(ループ内 fetch、依存の無い連続 await の直列実行、同一データの重複取得、キャッシュ / メモ化の不在)
- [ ] 機密の露出がないか(機密値の console 出力 / URL クエリ文字列への付与 / レスポンスへの不要な内部フィールド混入)
- [ ] 既存テストへの影響が洗い出され、必要な更新が計画されているか

## レビュー結果の扱い

1. reviewer は指摘を「箇所(ファイル:行)+ 種別 + 重大度」で報告する。
2. team-lead が各指摘を実コードで裏取りし、valid / invalid(false positive・理由記録)/ needs-user にトリアージする。
3. valid のみ implementer(小規模は team-lead)が反映し、再レビューする。
4. **同一指摘が 2 回連続で残存 → ユーザー確認。5 ラウンド超 → コスト警告。** ループを打ち切る。
