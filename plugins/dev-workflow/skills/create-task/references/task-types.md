# タスク種別ガイド(Phase 1 で判定・Phase 2 で適用)

要求を分析したら、まず該当する種別を判定する。種別ごとに「設計観点 / テスト手法 / 必須節 / 添える図」が変わる。**複合タスクは該当する全種別の要点を合成する**(例: 新機能+DB 変更 → 操作カバレッジ+ER 図+ロールバック)。種別が 3 つ以上に跨る場合は分割を検討する。

図の記法は doc/README.md の図規約(Mermaid / ASCII)に従い、**変更に関わる差分だけを図にする**(全体図は doc/ の責務。タスクの図は対象範囲の拡大図)。

## 種別マトリクス(早見表)

| 種別 | 標準テンプレに追加する必須節 | 添える図 | テストの型 | 実動確認(do-task Phase 5.5) |
|---|---|---|---|---|
| 新機能 | 操作カバレッジ表・E2E シナリオ | 業務フロー図(+UI があれば画面遷移図) | 正常系+主要な準正常系 | ハッピーパスを実際に操作(UI = ブラウザ / CLI = 実行) |
| 機能変更 | 操作カバレッジ表・既存挙動の回帰確認 | 変更部分の業務フロー図 | 回帰+変更点 | 変更フロー+隣接する既存フローの操作 |
| バグ修正 | 再現手順・原因分析・横展開確認 | (経路が複雑な場合)不具合経路のシーケンス図 | **回帰テスト先行** | **元の再現手順を再実行し、再現しないことを確認** |
| DB スキーマ変更 | ロールバック手順・型伝播の全径路 | **ER 図(変更部分の before/after)** | migration 往復+データ整合 | migration up→down→up の往復+データ確認クエリ |
| API 実装・変更 | 契約定義(入出力・エラー形) | **シーケンス図** | 正常 / 準正常 / 異常の 3 層 | 実リクエスト(正常+代表異常)を投げて入出力を確認 |
| UI 実装・変更 | 状態網羅(loading / error / empty) | **画面遷移図+ASCII ワイヤーフレーム** | 表示状態+操作シナリオ | 対象画面で 4 状態(loading/error/empty/正常)を目視 |
| 状態機械を持つ機能 | 遷移表・不正遷移の防御 | **状態遷移図** | 全遷移+不正遷移の拒否 | 代表遷移+不正遷移を実際に操作 |
| パフォーマンス | 計測方法・現状値・目標値 | (必要時)処理フロー図 | before / after の計測比較 | 計測を実行し before/after の数値を並べる |
| リファクタ(--refactor) | 挙動維持の検証方法・before/after メトリクス・段階ゲート([refactor-analysis.md](refactor-analysis.md)) | (再編時)before/after 構成ツリー | 既存テスト緑維持+テスト無し箇所は**特性テスト先行** | 主要フローの操作(挙動が変わっていないこと) |

**調査は種別に含めない**: 要求がまだ調査段階(何を作るか未確定・実現可能性が不明)の場合、タスク化せず先に調査を行う(SKILL.md Phase 1 の入口ガード参照)。調査の結論を doc/04 の ADR に記録してから、それを入力として create-task に戻る。

## 各種別の要点

### 新機能
- 設計: **参考実装の踏襲を最優先**(新規パターンの発明は、既存に該当が無いと確認できたときの最終手段)。操作カバレッジ表で CRUD・一覧・検索の扱いを漏れなく決める
- テスト: ハッピーパスの E2E シナリオ(または結合テスト)を完了条件に含める

### バグ修正
- 構造: ① 再現手順(確実に再現する最小手順)→ ② 原因分析(表層でなく根本原因)→ ③ 修正 → ④ 横展開確認(同じ原因のバグが他所に無いか Grep)
- テスト: **修正前に「落ちる」回帰テストを先に書き、落ちることを確認してから修正する**。これをチェックリストの先頭に置く

### DB スキーマ変更
- 影響: 型伝播チェーンの全径路(schema → 生成型 → server → API → client の手動更新点)を対象ファイル表に列挙する
- 図: 変更するテーブル+直接関係するテーブルだけの ER 図を before / after で(全体 ER は doc/04 の責務)
- 必須: ロールバック手順(migration の戻し方。不可逆ならバックアップ手段とリスク受容の確認)

### API 実装・変更
- 契約: エンドポイント・入出力スキーマ・エラーレスポンスの形を「技術的考慮事項」に明記する(実装前に契約を固定する)
- 図: 主要ユースケースのシーケンス図(クライアント → API → サービス → DB)
- テスト: 正常系 → 準正常系(バリデーションエラー・権限なし)→ 異常系(依存障害)の 3 層の順で書く

### UI 実装・変更
- 状態: **loading / error / empty / 正常の 4 状態**を実装タスクに含める(正常だけ作って終わりにしない)
- 図: 画面遷移図(`flowchart LR`)+対象画面の ASCII ワイヤーフレーム(高忠実度が必要なら Figma リンク)
- 観点: 既存デザインシステム・コンポーネントの再利用(参考実装節に記載)、a11y(キーボード操作・alt・コントラスト)

### 状態機械を持つ機能
- 図: `stateDiagram-v2` で全状態・全遷移を書く。**定義外の遷移要求をどう防ぐか**(ガード・エラー返却)を必ずタスク化する
- テスト: 正当な全遷移+不正遷移の拒否

### パフォーマンス改善
- 完了条件は数値で: 「計測方法(ツール・手順)/ 現状値 / 目標値」を明記し、before / after の計測をチェックリストに含める
- **憶測で最適化しない**: 計測 → ボトルネック特定 → 対処の順(stack-research にベストプラクティスがあれば参照)

### リファクタ(--refactor)
- **対象発見型**: 要求に対象が無ければ [refactor-analysis.md](refactor-analysis.md) の定量スキャン+スコアリングで候補を選定し、ユーザーに確認してからタスク化する
- **挙動維持が絶対条件**: 公開契約を変える必要が出たら「機能変更」として別タスクへ切り出す。検証方法(既存テスト維持 / **特性テスト先行** / 実動確認手順)を対象ごとにチェックリストへ
- **効果の定量化**: before メトリクス(行数・any 数・重複数)を現状分析に記録し、after 比較を完了条件に含める
- **段階ゲート**: 論理的なまとまり単位で Phase を割り、各 Phase 末尾に品質ゲート実行を置く(原因特定可能性の維持)
- テスタビリティ改善の手法は [testability.md](testability.md)、レビュー観点の素材は [refactor-review-checklist.md](refactor-review-checklist.md)
