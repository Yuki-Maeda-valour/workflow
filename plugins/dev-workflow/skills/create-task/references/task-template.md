# タスク MD 標準フォーマット

`task/進行中_{タスク名}.md` はこの構成で生成する。「(任意)」の節は該当するときだけ含める。
このフォーマットは /do-task が機械突合(チェックボックス集計・スコープ縮小検出)に使うため、章立てとチェックボックス記法を崩さないこと。

```markdown
# {タスク名}

> **ステータス**: 🚧 進行中
> **作成日**: {YYYY-MM-DD}
> **関連**: {PR / Issue / 会話の参照(あれば)}

## 概要

{何を・なぜ。2〜5 行。ユーザー要求の原文の要点も残す}

## 図解(種別に応じて。無関係な種別では本節ごと省略)

{references/task-types.md のマトリクスに従い、mermaid コードブロック(UI のワイヤーフレームは text コードブロックの ASCII)で
 「変更に関わる差分だけ」を図にする。例: DB 変更 = 変更テーブルの ER 図 before/after、API = 対象ユースケースのシーケンス図、
 UI = 画面遷移図+ワイヤーフレーム、状態機械 = 状態遷移図、業務フロー変更 = flowchart。記法は doc/README.md の図規約に従う}

## 現状分析

{実コード確認済みの現状。推測禁止}

### 対象ファイル

| ファイル | 現状 | 変更内容 |
|---|---|---|
| `src/...`(:行番号) | {現状の実装} | {何をするか} |

### 維持すべき制約(任意)

{既存の公開契約・挙動で壊してはいけないもの}

## 参考実装(踏襲するパターン)

| パターン | 参照先 | 何を踏襲するか |
|---|---|---|
| （例: 一覧+詳細画面の構成） | `src/...`(:行) | （構成・命名・エラー処理・テストの書き方） |

（該当が無い場合: 「該当なし・新規パターン」と明記し、設計判断を「技術的考慮事項」に書く）

## スコープ

- {今回やること}

## スコープ外

- {やらないこと} — 契約整合: {受理側を塞ぐ必要の有無と対応}

## 実装タスク

### Phase 1: {フェーズ名}

- [ ] **Task 1.1**: {タスク名}
  - **ファイル**: `{パス}`
  - **作業内容**: {具体的に。完了を確認できる粒度で}
  - **注意事項**: {既存パターンへの追従・落とし穴}

### Phase 2: {フェーズ名}

- [ ] **Task 2.1**: ...

## 操作カバレッジ(任意: エンティティ操作を変更する場合)

| 操作 | スコープ内 | 対応 / 整合確認 |
|---|---|---|
| 作成 / 参照 / 更新 / 削除 / 一覧 / 検索 | ○/×/− | {×の場合、不整合が生じない根拠} |

## テスト計画

| テスト | 種別 | 確認のみ / 更新必要 | 回帰検知条件 |
|---|---|---|---|
| `{テストファイル}` | unit/integration/e2e | {分類} | {何が壊れたら検知できるか} |

## 技術的考慮事項(任意)

{設計判断・代替案・パフォーマンス/セキュリティ観点}

## ロールバック手順(破壊的変更を含む場合は必須)

{migration・データ変換・削除を元に戻す具体手順。戻せない操作がある場合は「不可逆: {内容}」と明記し、実行前バックアップ等の代替策を書く}

## チェックリスト

- [ ] 実装タスクが全て完了している
- [ ] {format コマンド} が通る
- [ ] {check / lint コマンド} が通る
- [ ] {typecheck コマンド} が通る
- [ ] {test コマンド} が通る
- [ ] {build コマンド}(build_optional でない場合)
- [ ] {domain_risks 由来の固有検証(あれば)}

## 完了条件

1. {**実行可能な確認手順**として書く。「正しく動く」ではなく「① {起動・操作} ② {期待される観察結果}」の再現可能な形式。
   種別の実動確認(task-types.md の列)を反映する。例: バグ修正 → 「再現手順 X を実行し、エラーが発生しないこと」}
2. ...

## 追加修正記録

{/do-task が追記する欄。作成時は空}
```

## 記法の規約

- チェックボックスは `- [ ]` / `- [x]` のみ(全角・別記号を使わない)。/do-task が `grep -cE '^\s*- \[(x|X)\]'` で集計する
- 品質ゲートコマンドはプレースホルダのまま残さず、profile / scripts 検出で解決した実行形(例: `pnpm check`)を書く
- 「段階的に実施」「後続タスク」「今回はスコープ外」等の表現を実装タスク内に書かない。スコープ外にするものは必ず「スコープ外」節に契約整合とセットで書く
