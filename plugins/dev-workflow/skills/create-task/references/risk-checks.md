# ドメインリスク別チェック(profile: domain_risks)

profile の `domain_risks` に列挙されたキーに応じて、該当節の観点を Phase 1.5 とタスク MD のチェックリストに反映する。ここに無いドメインは地雷リスト(profile の `known_facts_ref`、既定は `doc/05_operations.md` の「引き継ぎ・地雷」節)と CLAUDE.md の記述から同様の観点を組み立てる。

## e2e-data — 共有テストデータの汚染防止

- E2E テストが前提とする共有アカウント・シードデータを特定し、タスクがそれを変更・削除しないか検証する
- テストデータを操作する場合は専用データを新設し、既存データへの影響ゼロをチェックリストに含める
- 並列実行される E2E との干渉(同一レコード更新)を考慮する

## i18n — 翻訳キー整合

- 翻訳ファイル(例: en / ja / default)の全ロケールでキーが揃うことをタスク化する(1 ファイルだけ更新して他を忘れるのが典型事故)
- キーの追加・削除は使用箇所の Grep 突合とセットにする
- ハードコード文言の混入をチェックリストに含める

## web3 — コントラクト連携

- ABI・型の再生成(codegen)タスクを忘れない(コントラクト変更 → 生成物更新 → フロント型整合)
- チェーン ID・ネットワーク設定への影響を確認する
- ウォレット接続フロー・署名 UX の回帰確認をテスト計画に含める

## zenstack / ORM スキーマ権限

- スキーマのアクセスポリシー(@@allow / @@deny 等)変更時、影響を受ける全ロールでの動作確認をテスト計画に含める
- 生成クライアントの再生成タスクを含める
- 権限エラーの返し方がフロントの期待(エラーハンドリング分岐)と整合するか確認する

## salesforce / 外部 SaaS 連携

- 外部 API のフィールドマッピング(命名の罠・レガシー命名)を実コードで確認してから対象ファイル表に書く
- 同期方向(片方向 / 双方向)と失敗時のリカバリをタスクに含める
- 外部側に反映されるデータの検証手順を完了条件に含める

## db-migration — 手動管理 DB / migration 欠落

- migration で管理されていないテーブルの有無を確認し、ある場合は SQL 手順を別タスクとして明記する(自動 migration できると仮定しない)
- 本番 / ステージングの DB 名・接続先の取り違え防止をチェックリストに含める
- 破壊的変更(カラム削除・型変更)はロールバック手順とセットにする
