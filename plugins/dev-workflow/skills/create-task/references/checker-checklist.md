# checker 検証観点(create-task Phase 3)

checker(Explore)はタスク MD を以下の観点で検証し、指摘を「対象箇所 / 問題 / 深刻度(blocker・major・minor)/ 提案」の形式で返す。

## 1. 実コード整合

- [ ] 現状分析に書かれたファイル・行番号・シンボルが実在するか(抜き打ちで数件、実ファイルを開いて確認)
- [ ] 「〜のはず」「おそらく」等、裏取りのない記述がないか
- [ ] 対象ファイル表に漏れがないか(タスク本文に登場するのに表にないファイル)

## 2. タスクの実行可能性

- [ ] 各タスクが「何をどのファイルにするか」まで具体化されているか(実装者が迷わない粒度)
- [ ] タスク間の依存順序が Phase 構成と矛盾していないか
- [ ] 「対応する」「整理する」「適切に処理する」等の曖昧語が残っていないか
- [ ] 実装タスク 10 個超・Phase 4 つ以上なら、分割が検討されたか(検討結果の明記があるか)
- [ ] 「参考実装」節があるか(類似パターンのパス、または「該当なし・新規パターン」の明記と設計判断)
- [ ] 破壊的変更(migration・データ変換・削除)を含むのに「ロールバック手順」節が無い → **blocker**
- [ ] タスク種別に応じた必須節・図が揃っているか(task-types.md と突合: バグ修正 = 再現手順+回帰テスト先行 / DB = ER 図+ロールバック / API = シーケンス図+ 3 層テスト / UI = 画面遷移+状態網羅 / 状態機械 = 状態遷移図 等)
- [ ] 図解節の図が「変更差分のみ」で、doc/README の図規約(記法・方向・日本語ラベル)に従っているか

## 3. スコープと契約

- [ ] スコープ外項目に契約整合の検証が書かれているか(受理するが処理されない不整合の有無)
- [ ] 公開契約(API・スキーマ・Props・イベント)を変えるタスクで、呼び出し元の更新タスクが漏れていないか
- [ ] 実装タスク内にスコープ縮小の温床表現(「段階的に」「後続タスク」「一旦」)が紛れていないか

## 4. 型伝播とデータフロー

- [ ] 型伝播チェーン上の手動更新ポイント(型注釈のハードコード・変換関数・バリデーション・モック)がタスク化されているか
- [ ] DB スキーマ変更がある場合、migration 生成・適用のタスクがあるか

## 5. テスト計画

- [ ] 契約変更に関わるテストが「更新必要」に分類されているか(「確認のみ」への逃げがないか)
- [ ] 回帰検知条件が「何が壊れたらこのテストが落ちるか」を説明できているか
- [ ] 新規機能に対する新規テストの要否が判断されているか

## 6. チェックリストと完了条件

- [ ] 品質ゲートコマンドがこのプロジェクトに実在するか(package.json scripts 等と突合)
- [ ] 完了条件が客観的に検証可能か(主観語「きれいに」「正しく」だけの条件がないか)
- [ ] domain_risks 由来の固有検証が反映されているか

## 7. フォーマット規約

- [ ] task-template.md の章立て・チェックボックス記法(`- [ ]`)に従っているか
- [ ] ファイル名が `進行中_{タスク名}.md` か
