# リファクタ分析ガイド(--refactor / リファクタ種別のとき)

リファクタは「何をしたいか」ではなく「**どこが悪いかを探す**」から始まる対象発見型のタスク。researcher はこのガイドの定量スキャン → スコアリングで対象を発見・選定する(対象パス・`--area` 指定があればその範囲に絞る)。候補が複数あれば選定結果(スコア付き)を提示し、どれをタスク化するかユーザーに確認する。

## 大原則: 挙動維持

外部から見た振る舞い・公開契約(公開 API / props / 関数シグネチャ / エンドポイント / 返却値の形)を変えない。UI は見た目・操作結果を同一に保つ。**契約を変える必要が出たら、それはリファクタではなく「機能変更」として別タスクに切り出す**。

## 1. 定量スキャン(メトリクス閾値)

閾値超えの箇所を抽出する。React 固有行は該当スタックのみ適用。Serena があればシンボル単位(`get_symbols_overview` / `find_symbol`)、無ければ Glob / Grep / Read で。**除外**: テスト・生成物・node_modules・ビルド成果物・設定ファイル・機密ファイル。

| メトリクス | 閾値 | アクション |
|---|---|---|
| ファイル行数 | > 200 行 | 責務ごとに分割 |
| 関数/メソッド行数 | > 50 行 | 小関数へ抽出・早期リターンで平坦化 |
| 引数の数 | > 7 個 | オブジェクト引数化・分割 |
| ネストの深さ | > 4 段 | 早期リターン・関数抽出で平坦化 |
| 重複コード | 3 箇所以上 | 共通化(関数/コンポーネント/モジュール) |
| インライン計算・変換 | > 3 箇所/ファイル | 純粋関数へ抽出 |
| マジックナンバー/文字列 | > 2 箇所/ファイル | 定数へ外部化 |
| 型の緩さ(any 等) | > 0 箇所 | 具体型に置換 |
| (React) useState/useEffect | > 5 個 | カスタムフック抽出 |
| (React) JSX return | > 100 行 | サブコンポーネント化 |
| (React) JSX 前のロジック行数 | > 50 行 | フック/純粋関数へ抽出 |

## 2. スコアリングで対象選定

各候補を加点し、上位(目安 5 件以内)をタスク化候補にする:

| スコア基準 | 加点 |
|---|---|
| 行数 150 行超 | +2 |
| 行数 300 行超 | +4(上記と累積) |
| 関数/コンポーネント 5 個超 | +2 |
| 責務の混在(UI + ロジック + データ取得 等) | +3 |
| コード重複パターン | +3 |
| 型の緩さ(any / as 多用) | +1(箇所ごと) |
| 巨大関数(50 行超) | +2 |
| プロジェクト規約違反 | +2 |

全体探査(対象未指定)では、スキャン対象ディレクトリ群を複数の Explore に並列で割り当ててよい。

## 3. アンチパターン検出

Grep(または Serena `search_for_pattern`)で探す。`doc/06_stack-notes.md` があれば記載のバージョン固有アンチパターンも検出対象に加え、ベストプラクティスを改善方針の参考にする。

- 三項演算子の連鎖: `\? .* : .* \?` / switch 文: `switch\s*\(`
- 緩い型: `:\s*any[^a-zA-Z]` と `as\s+any`
- 表示層内の数値計算: `Math\.` / 直接フェッチ: `fetch\(` `axios\.` / 日付処理の散在: `new Date\(`
- ループ内の fetch / クエリ(N+1 の温床): `for` / `map` ブロック内の `fetch\(` `await .*(find|query|select)`
- 依存の無い連続 await(並列化余地): 連続する `await` 行を検出し、依存の有無を目視判定(無ければ `Promise.all` 化)
- 機密のログ / URL 露出: `console\.(log|error)\(.*(token|secret|password|apiKey)`、URL クエリ文字列への機密付与

## 4. 改善パターン(タスクの「作業内容」に使う語彙)

| パターン | 適用条件 | 方針 |
|---|---|---|
| ファイル/コンポーネント分割 | 200 行超・責務混在 | 責務ごとに分割。50 行未満への断片化は避ける |
| ロジック抽出 | 巨大関数・巨大フック | 純粋ロジックは lib、副作用は Hook/サービスへ(詳細: [testability.md](testability.md)) |
| 共通化 | 同一パターン 3 箇所以上(2 箇所でも増加見込みなら可) | 見た目が似ても責務が違えば共通化しない |
| ディレクトリ再編 | 散在・機能凝集度の低下 | colocation・feature-based へ統一。全 import 更新+空ディレクトリ削除 |

テスタビリティ改善(純粋関数抽出 / 副作用分離 / 3 層分離 / 型厳密化 / 定数外部化)の詳細と before/after は [testability.md](testability.md)。

**スタック別の読み替え**: 3 層分離・Container/Presentational は React 系の型。バックエンドはルート/サービス/リポジトリの責務分離(依存は外→内の一方向)に、非 JS は純粋関数抽出・責務分離・定数外部化・早期リターン平坦化(言語非依存の核)に読み替える。フレームワーク固有の推奨構造は権威参照(CLAUDE.md / doc)と既存パターンに合わせる。

## 5. リファクタタスク MD に必ず入れるもの

- **before メトリクス**: 選定根拠の数値(行数・any 数・重複数など)を現状分析に記録。**完了条件に「after メトリクスを取得し before と比較して報告する」を含める**(効果の定量化)
- **挙動維持の検証方法**(対象ごとに 1 つ以上をチェックリストへ):
  1. 既存テストが対象を覆っている → そのテストの緑維持
  2. **テストが無い箇所を大きく動かす → 特性テスト(characterization test)を先に書いて現在の挙動を固定してからリファクタする**(実装タスクの先頭に置く)
  3. テスト化が難しい → 実動確認手順(主要フローの操作と期待される観察結果)を完了条件に明記
- **段階ゲート**: 実装タスクを「論理的なまとまり」単位の Phase に割り、**各 Phase の末尾に品質ゲート実行のタスクを置く**(全部変えてから検証すると原因特定が困難になるため)
- **レビュー観点**: [refactor-review-checklist.md](refactor-review-checklist.md) の 6 カテゴリから該当項目をチェックリストに反映する(機能保全・断片化回避・import 全更新・barrel 方針など)
- ロールバック手順は原則不要(挙動維持なので git revert で戻る)。ディレクトリ再編を含む場合のみ「git mv の履歴が追えること」を注記
