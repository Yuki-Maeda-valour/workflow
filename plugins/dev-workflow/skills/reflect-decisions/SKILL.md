# reflect-decisions — 決定事項のドキュメント反映

議事録・音声文字起こし・チャットログ・口頭指示のメモ・仕様資料といった「人間の合意・決定」を含む原資料を、doc/ の要件定義・設計・運用文書へ反映する。/update-doc が「実コード → ドキュメント」の同期であるのに対し、本 skill は「人の決定 → ドキュメント」の同期を担う。init-project 直後の空テンプレートを最初の打ち合わせ内容から起票する用途と、プロジェクト進行中の要件・設計変更を追従させる用途の両方に使う。

処理の流れ: **把握(Phase 1)→ 抽出(Phase 2)→ 精査(Phase 3)→ 適用(Phase 4)→ 適用後検証(Phase 5)→ 報告(Phase 6)**。

## 原則

1. **抽出即反映をしない**。抽出 → 精査(裏取り・レビュー・確認)→ 適用の順を必ず守り、精査を通過するまで doc を一切書き換えない
2. **出典裏取り**。反映する内容は必ず入力(原資料)内の発言・記述に遡れること。原資料に無い要件・理由を推測で補って書かない(体裁を揃えるための整形・要約は可。意味の追加は不可)
3. **決定と案の峻別**。文字起こし・チャットログには検討途中の案・雑談・却下案・暫定合意が混ざる。ドキュメントに「決定」として反映するのは合意が読み取れるものだけ。読み取れないものは未決として報告に回す(勝手に `[決]` を付けない)
4. **矛盾は上書きせず対比提示**。既存 doc の記述と食い違う決定は、黙って書き換えず新旧対比で提示して確認を取る(`--yes` でも矛盾項目の確認は省略しない)
5. **正本規約を守る**。書き込み先は doc/ のみ(要件+タグ・設計判断 ADR・運用は常に doc/ が正本)。ソースコード・Serena メモリ・`doc/06_stack-notes.md`(/stack-research 管轄)は変更しない。**`[実]` タグは付与しない**(実コードでの動作確認を伴う昇格は /update-doc の責務)
6. **トレーサビリティと冪等性**。反映した各項目に出典(日付・会議名/資料名。原文を保存した場合は相対リンク)を残す。同一出典の反映済み記載がある項目は再反映しない(同じ資料を 2 回渡されても doc が壊れない)
7. **機密保護**。原資料にトークン・パスワード等の機密値が混ざっていても doc に書かない(存在と用途だけ記す)

## 入力と引数

| 引数 | 内容 |
|---|---|
| ファイルパス(複数可) | 議事録・文字起こし・チャットログ・仕様メモ等(md / txt / pdf 等。PDF は Read で直接読める)。音声ファイル自体は扱わない(文字起こし済みテキストを渡してもらう) |
| (引数なし) | 会話に貼り付けられたテキスト・直前の口頭指示を入力とする。それも無ければ「原資料の貼り付け or ファイルパス指定」を促す |
| `--analyze-only` | Phase 3 の精査結果(反映マップ)の提示まで(書き込みなし) |
| `--no-review` | Phase 3-3 の外部レビューを省略(自己裏取りとユーザー確認は行う) |
| `--yes` | 矛盾のない項目の適用確認をスキップ(矛盾項目・決裁者不明の決定・誤認識候補は常に個別確認) |
| `--max-review=<N>` | レビュー反復の上限(既定: 無制限+セーフティ) |

## Phase 1: 入力とプロジェクト状態の把握

1. `.claude/project-profile.yml` があれば Read し、`root` / `source_of_truth` を解決する(無くても続行)
2. **doc ディレクトリを検出**する: `doc/` → `docs/` の順で探す。統一構成(README 索引+ 01〜05)が無い場合は /init-project の先行実行を提案する(承諾されればチェーン実行してから続行。即席で 03 だけ作らない — 構成の設置は init-project の責務)
3. doc/README.md(索引)と 01_overview / 03_requirements / 04_design / 05_operations / 07_plan(あれば)を読み、現在の記述状態(空テンプレか・どこまで書かれているか・タグ凡例)を把握する。**01 の用語集は Phase 2 の誤認識検出でも使うため控えておく**
4. 入力を読み込む。**資料メタ情報**(日付・会議名/資料名・参加者・議題)を冒頭やファイル名から抽出する。日付が読み取れなければファイルの更新日時から推定し、ユーザーに確認する
5. **決裁者の推定**: 参加者の役割(決裁者=発注側・PM / 開発側 / その他)を資料の文脈から推定する。読み取れない場合は「どなたの合意をもって決定とみなすか」をユーザーに確認する(毎回聞かない — 同一プロジェクトで一度確認したら以後はその前提を使う)
6. **原文の保存(トレーサビリティの起点)**: 入力が貼り付けテキスト・リポジトリ外のファイルの場合、この時点でプロジェクト内への保存を提案する(既存の議事録・資料置き場があればそこへ。無ければ例: `doc/minutes/YYYY-MM-DD_<件名>.md`)。保存すると doc の出典を相対リンクにでき、レビュアーも原文をパスで照合できる。不要と言われたら出典は「(日付 会議名)」の文字列表記とし、報告に出典要約を残す
7. 長大な文字起こし(目安 2,000 行超)は分割して読む。複数ファイルは時系列順(古い資料 → 新しい資料)に処理する。**後勝ちの原則**: 後の資料の決定が前の決定を覆す場合も、**同一資料の中で**前半の決定が終盤に覆された場合も、最終状態だけを反映対象にする(覆った経緯は必要なら ADR の却下案・備考に残す)

## Phase 2: 抽出(原資料 → 構造化)

入力から反映候補を抽出する。**1 項目につき必ず「根拠(元の発言・記述の短い引用または行位置)」を付ける**。この段階では doc を書き換えない。

**合意度の判定基準:**

| 分類 | 手がかり(例) | 扱い |
|---|---|---|
| **決定** | 「〜で行きましょう」「では〜に決定」「それでお願いします」提案への明示的な同意・反対なく次の議題へ進んだ結論・資料上の確定記述 | 精査へ回す |
| **暫定決定** | 「一旦〜で行く」「とりあえず〜」「たぶん〜になる」条件付き・仮の合意 | 精査へ回す。反映時は備考に「暫定(日付)」を明記し、確定した決定と区別する |
| **未決・宿題** | 「持ち帰って検討」「次回までに」「〜さん確認お願いします」担当・期限だけ決まった項目 | doc に書かず報告へ(タスク化候補) |
| **却下案** | 「〜はやめておこう」「今回は見送り」比較検討の末に選ばれなかった選択肢 | 単独では書かず、対応する決定の「却下した代替案」(ADR)として添える |
| **雑談・言い直し** | 議題と無関係な話・同じ内容の繰り返し | 無視する |

- **決裁者チェック**: 「決定」と判定した項目のうち、決裁者(Phase 1-5 の役割)の同意が読み取れないもの(開発側だけの合意・一部参加者のつぶやき)は、決定として確定せず Phase 3-4 で「決定として扱ってよいか」を個別確認する
- **誤認識の検出**: 音声文字起こしは固有名詞を壊しやすい。用語集・doc・コード上の識別子と「近いが一致しない」語は誤認識を疑い、**自動修正はせず**補正候補(例: 「プリズム → Prisma ?」)として Phase 3-4 の確認に回す

**抽出する種別**(1 項目 = 種別+内容 1 行+根拠+合意度):

- 目的・背景・スコープの変更(やること / やらないこと)
- 機能要件・非機能要件の新規・変更・取り下げ
- 前提・制約の変化(外部依存・環境・予算・期限)
- 設計判断(技術選定・方式・トレードオフと、その理由・却下した代替案)
- 画面・業務フローの合意
- 運用・環境・体制の決定
- スケジュール・マイルストーン・見積もり・予算の合意
- プロジェクト固有の用語(一般用語と違う使い方をするもの)
- 未決事項・宿題(担当・期限があれば併記)
- 上記に当てはまらない決定も捨てずに拾い、「反映先なし」として Phase 3-2 へ回す

## Phase 3: 精査(反映前の検証 — まだ書き込まない)

抽出結果をそのまま適用せず、以下の 4 段階を通過したものだけを「反映案」として確定する。

**3-1. 自己裏取り**: 各項目の根拠(引用)を原文と突き合わせ、拡大解釈・文脈の切り取り・複数発言の混同が無いか再確認する。合意度の判定(決定 / 暫定 / 未決 / 却下)も見直し、確信が持てないものは未決へ降格する。

**3-2. doc/ との突合(反映マップ)**: 各項目を既存 doc と突合し、**新規**(既存に無い)/ **更新**(既存の詳細化・変更)/ **矛盾**(既存記述と食い違う)/ **既出**(既に書かれている、または同一出典の反映済み記載がある → 再反映しない)に分類したうえで、反映先を決める:

| 抽出種別 | 反映先 | 書き方 |
|---|---|---|
| 目的・背景・スコープ | 01(目的 / 背景 / スコープ / スコープ外) | `[決]` 行の追加・更新 |
| 用語 | 01 用語集 | 表に追加 |
| 機能要件(新規) | 03 機能要件表 | `[決]` で追加(将来スコープは `[将]`)。1 行 1 要件。暫定決定は備考に「暫定(日付)」 |
| 機能要件(変更) | 03 該当行 | 内容を更新し、備考に変更経緯と出典。**`[実]` の要件を変える決定はタグを `[決]` に戻し**、実装が未追従である旨を注記 |
| 要件の取り下げ | 03 該当行 | 行は残し備考に「取り下げ(日付)」。行の削除はユーザー承認時のみ |
| 非機能要件・前提 | 03(非機能 / 前提・制約) | `[決]` / `[前]` |
| 設計判断 | 04 の ADR 表 | 日付 / 決定 / 理由 / 却下した代替案を 1 判断 1 行。大きな判断は `04_design/adr-YYYYMMDD-<タイトル>.md` に切り出し |
| 画面・業務フローの合意 | 03 業務フロー / 04 画面設計 | 既存の図と乖離したら図も更新(Mermaid。記法は doc/README.md の図規約に従う) |
| 運用・環境・体制 | 05 | 該当節に追記 |
| スケジュール・マイルストーン・見積もり・予算 | 07(あれば) | 見積もり表・マイルストーン表・変更履歴に出典付きで追記。状態列は `確定` / `暫定`(備考に「暫定(日付)」)。期日が揃ったらガント図(Mermaid `gantt`)も更新。**07 が無い場合は下の「反映先の節が無い情報」として扱う** |
| **反映先の節が無い情報** | **無理に押し込まない** | Phase 6 の報告で内容と置き場の提案(例: 07 の設置・01 への節追加・別文書化)を提示し、ユーザーの判断に委ねる |
| 未決事項・宿題 | **doc に書かない** | Phase 6 の報告でタスク化候補として列挙 |

- **02(アーキテクチャ)・06(スタックノート)は反映対象にしない**: どちらも実態(実コード・実依存)が正本のため。技術選定・方式の決定は 04 の ADR に記録し、実装が済んだら /update-doc が 02 を追従させる(06 は /stack-research の管轄)
- **実装済みの記述と食い違う決定**: doc は決定に合わせて更新する計画とするが、コードには触れない。「実装が未追従」であることを備考と報告に明記し、/create-task を提案する
- 反映先の既存記述が「（〜を記述）」のガイド文のままの場合はガイド文の置き換え(初回起票)として扱う

**3-3. 外部レビュー(`--no-review` 以外)**: 反映案の段階で(doc を書き換える前に)多モデルレビューにかける。

1. **編成の規模適応**: 標準は**能力帯の異なる複数レビュアーを単一メッセージで並列 Agent 起動**(実行環境で利用可能なモデルから上位+標準を選ぶ。profile の `features.review_models` 優先。Claude Code の目安: opus + sonnet。エイリアスのみ・読み取り専用)。**反映案が少数(目安 3 件以下)かつ矛盾ゼロ**の場合はレビュアー 1 体(または観点を分けたセルフレビュー)に軽量化してよい。どの編成で実行したかを報告に明記する
2. レビュアーには**原文のパス**(Phase 1-6 で保存したファイル、または元のファイルパス)・既存 doc のパス・反映案の一覧を渡し、引用箇所は各自 Read で照合させる(長大な原文をプロンプトに丸ごと貼らない)。合格基準:
   - **出典整合**: 反映案がすべて入力中の発言・記述に遡れるか。拡大解釈・捏造が無いか
   - **峻別**: 案・未決・却下を `[決]` として反映しようとしていないか。暫定に「暫定」が付いているか。`[実]` を新規付与していないか
   - **反映先の妥当性**: 反映先文書・節・タグの選択が反映マップと doc 統一構成に合っているか
   - **既存 doc との整合**: 見落とした矛盾・重複・既出が無いか
3. team-lead が各指摘を**入力原文と doc の実ファイルで裏取り**し、valid / invalid / needs-user にトリアージする(盲信しない。invalid は理由を記録)
4. valid を反映案に修正 → 再レビュー。**全レビュアー PASS(valid 0 件)で合格**
5. セーフティ: 同一指摘 2 回連続残存 → ユーザー確認 / 5 ラウンド超え → トークンコスト警告 / `--max-review` 到達 → 状況報告
6. 記録: `.claude/reviews/reflect-decisions-iter{N}.md`
7. **環境縮退**: サブエージェント機構が使えない環境では、観点(出典整合 → 峻別 → 反映先 → 整合)を分けたセルフレビューに縮退し、どの段階で実行したかを報告に明記する

**3-4. ユーザー確認**: 精査済みの反映一覧(対象文書 / 分類 / 内容 / 出典)を提示する。以下は `--yes` でも必ず個別確認する:

- **矛盾項目**: 既存記述との新旧対比を添えて
- **決裁者の同意が読み取れない「決定」**: 決定として扱ってよいか
- **誤認識の補正候補**: 補正して反映してよいか

`--analyze-only` はここで終了(反映推奨リスト+本実行の案内)。

## Phase 4: 適用(各ドキュメントへ)

1. **承認された反映案のみ**を 1 ファイルずつ適用する。精査で確定した内容から変えない(適用中に気づいた追加変更は勝手に混ぜず、Phase 6 で報告して次回に回す)
2. 各追記・変更に出典を残す — 原文を Phase 1-6 で保存した場合は保存先への相対リンク、そうでなければ「(YYYY-MM-DD 会議名/資料名)」形式。ADR は日付列に決定日を入れる
3. **索引の更新**: 内容を変更した文書について doc/README.md の状態絵文字・最終更新日を更新する

## Phase 5: 適用後の機械検証

1. **反映案との突合**: 適用結果(diff)と承認済み反映案を突き合わせ、反映漏れ・案に無い変更の混入が無いことを件数と内容で確認する
2. **体裁と整合の最終確認**: タグ凡例(`[実][決][将][前]`)・1 行 1 要件・ADR 表の体裁、更新した文書間(01 ↔ 03 ↔ 04 ↔ 05 ↔ 07)の矛盾(例: 03 のスコープ変更と 07 の見積もり前提のずれ)、記載パス・リンク(出典の相対リンク含む)の実在を確認する
3. `.claude/grasp.md` があれば削除する(doc の前提が変わったため、次回の /understand-project が再把握して作り直す)

## Phase 6: 報告とチェーン

1. **反映サマリ**: 文書別の変更点 / 新規・更新・矛盾解決・既出スキップの件数 / 見送った項目とその理由(合意が読み取れない・精査で降格・決裁者未確認等)/ レビュー編成(標準・軽量・縮退のどれで実行したか)
2. **未決事項・宿題の一覧**(担当・期限付き)。決まらなかったことが次に何を待っているかが分かる形で
3. **反映先が無かった決定**(スケジュール・体制等)の内容と置き場の提案
4. **チェーン提案**:
   - 実装が必要な決定・宿題がある → 「/create-task でタスク化しますか?」を AskUserQuestion で確認(Yes ならチェーン実行。**タスク化する項目ごとに決定内容・出典(リンク)・期限・担当を引き継ぐ**)
   - 概要・技術などコード理解系の記述に波及する変更があり、正本がメモリ(`source_of_truth: serena`)のプロジェクト → /update-doc でメモリへの追従を案内

## 最終ゲート(完了報告前セルフチェック)

- [ ] 精査(Phase 3)を通過していない内容で doc を書き換えていない(抽出 → 精査 → 適用の順を守った)
- [ ] 反映した全項目に入力内の根拠(出典)がある。推測で補った要件・理由が無い
- [ ] 未決・案・却下を `[決]` として書いていない。暫定決定に「暫定(日付)」を明記した。`[実]` を新規付与していない
- [ ] 決裁者の同意が読み取れない「決定」・誤認識の補正候補を、確認なしで反映していない
